CFNA CONCURRENT IMPLEMENTATION PLAN
Encoding: UTF-8
Format: Plain text
Repository: https://github.com/LMMinier/nueronce

MISSION

Turn the current CFNA prototype into a reproducible, CPU-first, compact foundation-model research system.

PRIMARY TARGET

Build CFNA Research Model A with these properties:

1. 10M to 40M parameters.
2. Trains on a real, balanced corpus.
3. Produces coherent text.
4. Follows simple unseen instructions.
5. Saves and resumes training reliably.
6. Performs true incremental streaming inference.
7. Uses retrieval for external knowledge.
8. Is compared against matched baselines.
9. Reports CPU time, RAM use, and inference speed.
10. Exposes ablations for every major mechanism.
11. Can be reproduced from a clean repository clone.

NON-NEGOTIABLE RULES

1. CPU is the reference runtime.
2. GPU support is optional and must not dictate the architecture.
3. Do not hide a pretrained language model inside CFNA.
4. Open models may only be used as explicit teachers, external renderers, retrievers, or evaluators.
5. Do not train on unverified web scrapes.
6. Do not use arbitrary remote code execution for datasets.
7. Do not load untrusted pickle, PT, or PTH files.
8. Every subsystem must be labeled as one of:
   - REAL / TRAINABLE
   - REAL / HEURISTIC
   - INTERFACE ONLY
   - PLANNED
9. Never report training loss as proof of generalization.
10. Never claim superiority without matched baselines, multiple seeds, and ablations.
11. Do not scale to 337M until Research Model A passes all gates.
12. When a hypothesis fails, record it and simplify or remove it.

CURRENT REPOSITORY BASELINE

The repository already contains:

- Trainable byte-level CFNA core.
- Dynamic patching.
- Typed recurrent state.
- Hybrid SSM and attention paths.
- Retrieval training path.
- Byte Transformer and pure SSM baselines.
- Public-domain corpus pipeline.
- Checkpoint training.
- Chat-style continuation wrapper.
- Heuristic planner and verifier.
- Real but untrained 337M constructor.

Current known gaps:

- No true streaming inference.
- No true conditional execution.
- Sparse attention still uses dense score computation.
- Typed channels are not proven semantically specialized.
- No mature instruction-tuning pipeline.
- Checkpoint resume is incomplete.
- No strong real-corpus benchmark.
- No proof of CPU efficiency advantage.

CONCURRENCY MODEL

Use five agents in parallel.

AGENT 1: TRAINING RELIABILITY
AGENT 2: CORPUS AND DATA
AGENT 3: STREAMING RUNTIME
AGENT 4: INSTRUCTION TRAINING
AGENT 5: EVALUATION AND CPU PROFILING

Each agent must:

1. Declare owned files.
2. Avoid editing files owned by another active agent.
3. Define acceptance tests before implementation.
4. Produce machine-readable metrics.
5. Use an isolated branch or worktree.
6. Merge only after tests pass.
7. Record unresolved limitations.

CORE FILE OWNERSHIP

Conflict-prone files:

- cfna/model.py
- cfna/blocks.py
- cfna/nn.py
- cfna/config.py

Only one agent may edit each core file at a time.

Prefer additive modules and adapters instead of simultaneous core rewrites.

Before editing a core file, an agent must record:

- File name.
- Symbols to change.
- Reason for change.
- Expected tests.
- Expected merge order.

LANE D: REAL LOCAL AND SPARSE COMPUTATION
=========================================

OWNER FILES

- cfna/nn_local.py
- cfna/nn_sparse.py
- cfna/nn.py
- tests/test_attention_complexity.py
- benchmarks/benchmark_attention.py

GOAL

Replace masked dense attention with operators that do not construct a full T by T score matrix.

IMPLEMENT THREE MODES

1. Sliding-window local attention.
2. Landmark or global-slot attention.
3. Retrieval-only cross-attention.

LOCAL ATTENTION RULE

For position t:

start = max(0, t - window)
keys = K[start:t+1]
values = V[start:t+1]
compute scores only over this local range

REQUIREMENTS

- No tensor with shape [batch, heads, sequence_length, sequence_length].
- Score-pair count scales approximately as sequence_length times window.
- Temporary tensor sizes are measured.
- Peak memory is measured.
- Wall-clock time is measured.

ACCEPTANCE TESTS

1. Local implementation matches masked-dense local attention numerically.
2. Pair count scales with T times window.
3. No full T by T score tensor is created.
4. Benchmark at T equal to 128, 256, 512, and 1024.
5. Memory growth is approximately linear for fixed window.

DELIVERABLES

- docs/ATTENTION_KERNELS.md
- benchmarks/attention_results.json
- migration plan from existing attention

DO NOT

Do not delete the existing implementation until parity tests pass.

GATE 5: ARCHITECTURAL EVIDENCE

Required:

- matched baselines
- multiple seeds
- ablations

One checkpoint must demonstrate:

- broad language modeling
- transfer to multiple tasks

================
STOP CONDITIONS
================

Stop scaling when any of these remain unresolved:

- held-out bpb no longer improves
- output stays incoherent after reasonable data and training increases
- streaming cannot match full inference
- typed channels show no specialization
- conditional routing saves no CPU wall-clock time
- retrieval does not improve grounded tasks
- CFNA is consistently worse than simpler matched baselines

When a mechanism fails:

1. Record the result.
2. Preserve the test.
3. Simplify or remove the mechanism.
4. Do not protect it because it was part of the original idea.
