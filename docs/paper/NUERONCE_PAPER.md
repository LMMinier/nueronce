# NUERONCE: A From-Scratch, Byte-Level, Provenance-Gated Language Model — Architecture, System Design, and an Honest Empirical Ledger

*A technical report on the "nueronce" codebase. Written for machine-learning researchers and for readers without a machine-learning background. Every major technical section contains a short paragraph marked "In plain terms:" that restates the section without jargon. Every number in this report is traceable to a file, test, or logged run in the repository; results that are weak, negative, or lost are reported as such rather than hidden.*

---

## 1. Abstract

We describe NUERONCE, a language model and surrounding system built entirely from primitive tensor operations — no pre-built transformer layers, no external state-space packages, and no learned tokenizer. NUERONCE reads and writes raw bytes. Instead of a fixed vocabulary, it learns where to cut a byte stream into variable-length "information units" using a learned boundary head: the discrete cut simply thresholds the learned boundary probability under minimum and maximum patch lengths, and the head is trained both by a self-supervised word-onset objective and by the language-model gradient itself. Units flow through a typed, multi-timescale recurrent memory and a hybrid cognitive core that merges a selective state-space path, local attention, sparse-global attention, and a retrieval path through a per-position adaptive router; a byte decoder then cross-attends to completed units to emit the next byte.

The design has four elements we believe are distinctive in combination: (1) learned dynamic byte patching driving a two-level byte model; (2) a native provenance/authority layer that separates *apparent authority* (learnable from text) from *authenticity* (provable only with cryptography), gating what evidence a response may rely on; (3) a second, from-scratch NumPy autograd backend that reproduces benchmark artifacts bit-for-bit across operating systems and Python versions; and (4) a pre-registered go/no-go gate methodology that ties each parameter rung to a numeric acceptance criterion before scaling.

Empirically, the largest *trained* model is an 11.1-million-parameter checkpoint. In a single A100 session on 2026-07-02 it fell from a held-out 4.515 bits per byte to 1.785 in 7,524 steps and 55 minutes, clearing a pre-registered gate at minute 51; continued training reached a best held-out 1.313 bits per byte with the first coherent greedy prose, and that checkpoint was then lost to an unpersisted runtime — an operational failure we report in full because it motivated an auto-backup mechanism. We also report a textbook overfitting case study, a falsified architectural hypothesis (typed-channel specialization), and a strong provenance result that holds zero poison acceptance across text-only attack families. We have *not* run any standardized external benchmark or an equal-compute transformer baseline; those omissions are stated plainly and deferred to Future Work.

**In plain terms:** This is a small, hand-built system that reads text as raw bytes rather than words, decides on its own how to chop text into chunks, and keeps a strict record of where every fact came from before it trusts it. The biggest version we actually trained is small by modern standards, and it learned to produce fluent-looking English in under an hour on one graphics card. We are careful throughout to separate what we measured from what we only designed, and we admit the experiments we have not yet run.

## 2. Introduction

The dominant recipe for large language models fixes a subword vocabulary in advance, embeds tokens into a single dense stream, and processes that stream with stacked dense-attention transformer blocks. This recipe is enormously successful, and the scaling report by Brown et al. (2020) is the canonical demonstration that capability emerges predictably with parameters and data. NUERONCE starts from a different set of design commitments and asks how far a very small model built on them can be pushed with a modest, license-clean corpus and honest bookkeeping.

The first commitment is *no borrowed machinery*. Every parametric layer in the model is constructed from parameter tensors and raw tensor math: linear maps, embeddings, root-mean-square normalization, gated feed-forward blocks, attention assembled from matrix multiplies and softmax, and a selective state-space recurrence written as an explicit scan. The deep-learning framework is used only as a tensor, autograd, and optimizer substrate. This is a deliberate constraint: it forces every architectural claim to be realized in code we can inspect, and it makes a *second*, independent backend feasible — a from-scratch NumPy autograd engine that runs the same architecture without the framework at all, which becomes the basis of an unusually strong reproducibility property (Section 6).

The second commitment is *bytes, not tokens*. The model's input and output alphabet is the 256 possible byte values, and it never touches a tokenizer. In place of fixed tokens, a perception encoder produces per-byte features and a boundary probability, and a dynamic patcher cuts the stream into variable-length units. Because the segmentation is learned (through an auxiliary boundary objective and a differentiable per-unit boundary feature), the model discovers its own units rather than inheriting them from a vocabulary fit on some other corpus.

The third commitment is *provenance as a first-class citizen*. Most systems treat "which sources do I trust" as a prompt-engineering problem outside the model. NUERONCE ships a typed cognitive contract and a cryptographic provenance gate that make trust an explicit, auditable computation: a document's *apparent authority* (how official it looks, learnable from text) is kept strictly separate from its *authenticity* (whether it is really from its claimed issuer, provable only by an Ed25519 signature), and high apparent authority without verified authenticity is never treated as trusted authority.

The fourth commitment is *pre-registered honesty*. The project maintains a live ledger of hypotheses and a parameter ladder (11M → 35M → 90M → 337M) in which each rung is gated behind a numeric acceptance criterion that is written down *before* the run. When a hypothesis fails — as the central "typed channels specialize" hypothesis did — it is recorded as failed and removed from the differentiation story rather than quietly retained.

This report documents the architecture (Section 4), the data and training procedure (Section 5), the engineering and system implementation (Section 6), the empirical results including the failures (Section 7), an honest discussion of strengths and limits (Section 8), and an itemized list of assumptions and future work (Section 9).

**In plain terms:** Most modern language models are built from big reusable parts, chop text into "words" using a fixed dictionary, and leave the question of which sources to trust outside the model. This project does the opposite on all three counts: it builds every part by hand, learns its own way to chop up raw text, and bakes source-checking directly into the system. It also holds itself to targets set in advance, and it writes down the experiments that did not work.

## 3. Related Work

NUERONCE sits at the intersection of three research threads, and the honest way to place it is to say exactly where it follows each thread and where it diverges.

**Byte- and character-level modeling.** The clearest antecedents are tokenizer-free models. ByT5 (Xue et al., 2022) showed that token-free byte models are viable and robust to noise at the cost of longer sequences. MegaByte (Yu et al., 2023) introduced a two-level design — a patch-level global model over segments plus a local byte model — to amortize the cost of long byte sequences; NUERONCE's split into a unit-level core and a byte-level decoder is squarely in this family. The Byte Latent Transformer (Pagnoni et al., 2024) is the closest in spirit: it replaces fixed patches with *dynamic*, entropy-driven byte patching, allocating compute where the next byte is hard to predict. NUERONCE's dynamic patcher is a sibling of that idea, but where BLT derives boundaries from a separately-trained entropy model, NUERONCE's cut is a threshold on a single learned boundary probability, trained two ways at once — by an explicit self-supervised word-onset loss (syntax bytes label where units begin) and by the language-model gradient flowing through a per-unit mean-boundary feature — while the discrete cut decisions themselves stay detached from the gradient. MambaByte (Wang et al., 2024) demonstrated byte-level modeling with a state-space backbone; NUERONCE also uses a selective state-space path, but as one of four routed paths rather than the sole mixer. A candid caveat: unlike these works, NUERONCE has not been evaluated on any standardized byte-level benchmark, so its placement here is architectural, not results-based.

**State-space and hybrid sequence models.** The selective state-space model Mamba (Gu & Dao, 2023) motivates NUERONCE's recurrent path, which implements an input-dependent selective scan by hand rather than importing a package. Hybrid designs that interleave state-space and attention — Jamba (Lieber et al., 2024) and the Griffin/RG-LRU family (De et al., 2024) — validate the general idea of mixing a recurrent mixer with attention. NUERONCE differs in *how* it mixes: rather than interleaving distinct block types by depth, every block computes a state-space output, a local-attention output, a sparse-global-attention output, and a retrieval output at once, then blends them per position with a small learned router. Whether this per-position fusion beats a plain interleave at equal compute is an open question we have not tested.

**Tokenizer-based frontier designs.** The large tokenized decoders — the GPT-3 family (Brown et al., 2020), the PaLM report (Chowdhery et al., 2022), and the compute-optimal scaling analysis of Chinchilla (Hoffmann et al., 2022) — are the backdrop against which NUERONCE's choices should be read. NUERONCE deliberately departs from the shared assumptions of that line: it has no subword vocabulary, no single homogeneous transformer stack, and no separation between "the model" and "the retrieval/trust layer." We borrow their *lessons* — in particular Chinchilla's warning that undertrained large models lose to well-trained small ones, which is why the parameter-ladder presets each pair a size with a minimum corpus expectation. We do not borrow their architecture, and we make no claim to their scale: the largest NUERONCE configuration that has ever seen a gradient is 11.1M parameters.

**Retrieval-augmented and provenance-aware systems.** The retrieval path draws on retrieval-augmented modeling in the RETRO lineage (Borgeaud et al., 2022), where a model cross-attends to retrieved neighbors. NUERONCE keeps retrieval at the byte level so exact identifiers and values remain copyable. Its genuinely distinctive contribution is not retrieval itself but the *provenance/authority layer* around it: a cryptographic gate (Section 6) that we have not seen combined with a from-scratch byte model in this way. The nearest conceptual neighbors are content-authenticity and signed-provenance efforts in security, not in language modeling.

Four elements distinguish NUERONCE from all of the above *in combination*, even though each has partial precedent: learned dynamic byte-patch composition driving a two-level byte model; a native provenance/authority layer that gates evidence; a dual-backend, bit-identical reproducibility harness; and a pre-registered acceptance-gate methodology.

### 3.1 How NUERONCE runs differently from the established LLM process

Because readers asked for the departures to be stated in one place, we contrast the established production process — the recipe shared, with variations, by the GPT/PaLM/LLaMA lineage — with what this codebase actually does, stage by stage. Every entry in the NUERONCE column is implemented and testable in the repository; none is aspirational. We emphasize before the table what the table is *not*: it is a list of verified process differences, not a claim of superiority — the equal-compute comparison that could support a superiority claim has not been run.

| Stage | Established process | What NUERONCE does instead (verified in code) |
|---|---|---|
| Text ingestion | A subword tokenizer is fit in advance; the model inherits its vocabulary and its failure modes (out-of-vocabulary artifacts, counting/spelling blind spots) | No tokenizer exists anywhere in the system. Raw bytes in, raw bytes out; a learned boundary head cuts the stream into variable-length units at run time, trained by word-onset self-supervision plus the language-model gradient |
| Sequence mixing | A single homogeneous stack of dense-attention transformer blocks (or, in newer hybrids, fixed interleaves of attention and state-space blocks by depth) | Every block computes four mixers at once — selective state-space scan, local windowed attention, sparse-global top-k attention, retrieval cross-attention — and a learned router blends them per position; a few physical blocks are reused over a configurable logical depth, so computation depth is decoupled from parameter count |
| Trust and retrieval | Source trust lives outside the model, in prompt engineering, retrieval-pipeline heuristics, or post-hoc moderation | A typed data contract carries each evidence item's apparent authority and cryptographic authenticity into the prompt itself; an Ed25519 signature gate refuses to treat official-looking text as trusted authority without verification, and this gate is benchmarked against attack families rather than assumed |
| Implementation | One implementation on one framework; correctness rests on the framework's test suite and the vendor's internal replications | Every layer built from primitive tensor operations, twice — once on a mainstream framework, once on a from-scratch NumPy autograd engine — with a benchmark that must reproduce bit-identically across backends, operating systems, and Python versions |
| Scaling decisions | Scale first, evaluate after; capability targets are typically set retrospectively against benchmark suites | A parameter ladder in which each rung's numeric acceptance criteria are written down before its training runs, and a rung cannot be started until the previous rung's gate is cleared |
| Reporting | Positive results are published; negative results and operational failures are rarely part of the record | A hypothesis ledger in which a central architectural hypothesis is recorded as falsified, an overfitting failure and a lost checkpoint are reported with numbers, and every claim in this report traces to a committed file or logged run |

Two of these rows deserve a caution. The per-position four-way routing is architecturally distinctive but empirically unproven against a plain interleave — that is exactly the kind of claim the missing baseline experiment exists to test. And the provenance row describes the symbolic layer; as Section 8 details, the neural model does not yet condition on the authority signals, so the difference is real at the system level but not yet inside the weights.

**In plain terms:** The standard way to build a language model is: chop text with a fixed dictionary, push it through one kind of repeated processing block, handle trust and safety outside the model, build everything once on one toolkit, scale up first and measure afterwards, and publish the wins. This project differs at every one of those six steps — no dictionary, four kinds of processing blended per position, trust checked cryptographically inside the data the model sees, everything built twice and required to agree exactly, targets set before training instead of after, and failures published next to successes. Different does not automatically mean better; the honest comparison experiments are still on the to-do list.

**In plain terms:** Other researchers have built models that read bytes instead of words, models that chop text into chunks on the fly, and models that mix two different kinds of memory. NUERONCE combines those ideas with two things that are unusual together: a built-in system for cryptographically checking where information came from, and a second, independent implementation that reproduces results exactly. It is small and unbenchmarked against the famous systems, so we compare designs, not scores.

**Summary.** NUERONCE is a byte-level, dynamically-patched, hybrid state-space/attention model in the lineage of ByT5, MegaByte, BLT, MambaByte, Mamba, Jamba, and Griffin, deliberately diverging from the tokenizer-based GPT-3/PaLM line, and extended with a provenance layer and a reproducibility discipline that have no direct precedent in language modeling.

## 4. Model Architecture

### 4.1 Design reasoning and the problems each stage solves

The architecture was assembled to answer a sequence of questions, each stage solving the problem the previous stage exposes. The first question is *what is the unit of processing?* Committing to raw bytes removes tokenizer bias and out-of-vocabulary failure but creates very long sequences. The response is a two-level structure: a cheap, local byte encoder feeds a learned patcher that compresses the stream into a small number of units, an expensive core reasons over units, and a byte decoder expands units back to bytes. This is the same amortization principle behind MegaByte and BLT, realized with NUERONCE's own components.

The perception encoder embeds each byte, then applies a stack of causal one-dimensional convolutions — a kernel-3 layer, a residual kernel-7 layer, and a residual dilated kernel-3 layer with dilation four — followed by root-mean-square normalization. Causality is enforced by left-padding, so no position ever sees the future. From the resulting per-byte features a small multilayer boundary head emits a per-byte boundary logit. This gives the model both a representation of local byte context and a first opinion about where units should begin.

The second question is *where to cut*. At run time the rule is deliberately simple: a boundary is cut when the learned boundary probability crosses a threshold and the current patch has reached a minimum length, or unconditionally when it reaches a maximum length. The sophistication is in how that single probability is *trained*, because a pure learned boundary is hard to train from scratch. A table of syntax bytes (whitespace and punctuation) supplies self-supervised targets — a boundary is labeled where a non-syntax byte follows a syntax byte, a word onset — so the head has a real signal from step one; the discrete cut decisions are taken under no-gradient so they never see this machinery. Two further subtleties make this trainable end to end. The discrete segmentation is computed under a detached, hard decision so the non-differentiable cut does not block gradients; separately, the *mean boundary probability inside each unit* is projected and added to that unit's representation, so the next-byte loss can still flow gradient into the boundary head — the model's main objective gets a vote on where boundaries belong, alongside the word-onset supervision. The total training objective is next-byte cross-entropy plus a weighted boundary cross-entropy, and bits-per-byte is reported as the cross-entropy divided by the natural logarithm of two.

The third question is *how should units be represented and remembered over time*. Pooled per-unit local features are projected to the model width and normalized, then passed through a typed, multi-timescale recurrent memory whose read-out is *added* to the unit representations. The memory maintains several typed channels — semantic, structural, goal, evidence, uncertainty, authority, and procedure — each with its own fixed retention timescale and an authority permission slot that can throttle writes. Per step the cell computes forget, write, and read gates and a candidate update; the new cell state decays the old state by the channel's retention times the forget gate and adds the authority-masked, write-gated candidate; the hidden read-out is the read gate times a nonlinearity of the cell. The channels are fused into single projections for speed while keeping per-channel retention and authority. This design is descended conceptually from the protected cell state of long short-term memory, with efficiency motivation from selective state-space models.

The fourth question is *what does reasoning over units look like*. A hybrid core block computes four things in parallel from the same normalized input: a selective state-space output, a local windowed-attention output, a sparse-global-attention output that keeps only each query's top-k causal keys, and a retrieval cross-attention output over any retrieved byte context. A small router produces a per-position four-way softmax and blends the four outputs; a gated feed-forward with residuals follows. Crucially, the routing at each position depends only on that position's own causal path outputs, so no future position can influence it. A few physical blocks are then reused over a mode-dependent *logical depth*, so the effective computation depth can exceed the parameter count — a memory-saving recurrence over shared weights.

The fifth question is *how to turn units back into bytes exactly*. The byte decoder embeds bytes, applies layers that first attend locally over byte history, then cross-attend to *completed* units under a causal byte-to-unit mask (a byte at position t may attend only to units strictly before its own segment), then optionally cross-attend to retrieved context, then a gated feed-forward. A final projection produces next-byte logits over the 256 byte values. The completed-unit masking is what keeps the whole two-level model a valid left-to-right autoregressive model.

The retrieval path deserves a specific note because it is engineered for *exact copying*, not fuzzy recall. Retrieved neighbor bytes are encoded so that each position exposes its left context as the matching key and an embedding of the byte itself as the copyable value; this makes retrieval an induction-style copy that can reproduce exact identifiers and numbers rather than paraphrases.

### 4.2 Configuration presets and the parameter ladder

The architecture is instantiated at four sizes, each a counted-by-construction configuration rather than a rounded label. The chat rung is 11.1M parameters and is the configuration the first real checkpoint used. The base rung is 34.4M parameters and is the recommended next step for a single consumer GPU. The larger base rung is 92.1M parameters for a mid-range GPU. The large configuration realizes the design's roughly 350M-parameter target and counts to about 337M; it constructs and forwards but has **never been trained** — its weights are random, and a scaling test only verifies it builds at the right size. The presets scale width, patch capacity, logical depth, and window sizes together, and their docstrings encode the Chinchilla lesson directly: scale data with parameters, and drop the learning rate roughly with inverse width.

**In plain terms:** The model works in five stages. First it reads raw bytes and forms a quick local impression of each one. Then it decides where to cut the text into chunks using a learned instinct for where words and ideas begin — an instinct it is taught by noticing where spaces and punctuation sit, and sharpened by the prediction task itself. Each chunk gets a memory that keeps seven kinds of running notes at different speeds. A reasoning core then thinks over the chunks using four different mechanisms blended together, and reuses the same parts several times to think more deeply without needing more parts. Finally, a decoder turns the chunks back into exact bytes one at a time. There are four size settings; the biggest one has been built but never trained, which we say plainly.

**Summary.** NUERONCE is a causal two-level byte model: a convolutional byte encoder and learned patcher compress bytes into units; a typed recurrent memory and a four-path routed hybrid core reason over units with weight reuse across logical depth; a byte decoder cross-attends to completed units and retrieved context to emit next-byte logits. It is trained with next-byte cross-entropy plus an auxiliary boundary loss, exists at four sizes from 11.1M to 337M, and only the smaller rungs have been trained.

## 5. Data and Training Procedure

### 5.1 Reasoning about data before compute

The project's data discipline is a direct reaction to two failure modes it had already suffered: overfitting a tiny instruction set, and poisoning a dataset with a single dominant register. The response is a staged, license-explicit corpus registry and a deterministic dataset builder that caps any single register.

The corpus registry enumerates 22 licensed sources, each with a page URL, file URL, license label, an intended training phase, and notes on whether it belongs to base pretraining or later instruction tuning. The sources span short narratives, educational expository text, filtered educational web text, deduplicated web text, educational code across roughly thirty languages, encyclopedic dumps, public-domain books, open textbooks, open scientific full text, mathematical/physics web text with formulas preserved, and — for later instruction tuning — human-assistant dialogue, an instruction set, grade-school math, and several multiple-choice science and commonsense sets. Two datasets are deliberately excluded for licensing reasons (a non-commercial science-questions set and an aggregated exam benchmark whose upstream terms are unclear), and the exclusions are annotated in the registry itself. This is a legal-hygiene decision as much as a modeling one: every source can be defended.

For instruction tuning, a deterministic builder assembles a mixed conversational supervised-fine-tuning dataset in the canonical prompt format. From the manifest, it draws candidates from every synthetic dialogue register, from prompt-aligned direct and grounded generators that exercise the evidence and plan blocks, and from a hand-written seed set; it validates and deduplicates, enforces a cap of at most 25% per register, splits and shards deterministically from a fixed seed, and runs a hash-based leakage check. The committed manifest records 57,343 accepted records after dedup and 54,685 after the register cap, with arithmetic and classification each held to 13,671 records — the direct remedy for a prior dataset that was 77% arithmetic. The dataset is *not* committed to version control; it regenerates bit-for-bit on any machine from the seed, which is both a storage decision and a reproducibility guarantee.

### 5.2 The training objective and the supervised-fine-tuning contract

Base pretraining minimizes next-byte cross-entropy plus a weighted boundary cross-entropy, and reports held-out bits per byte as the primary quality metric. Supervised fine-tuning uses a response-only loss mask: only the bytes of an assistant response, including its trailing end marker, contribute gradient; system text, user turns, evidence, plan blocks, and role markers are masked out. In multi-turn examples the mask is applied per turn. This contract is enforced by a test that asserts prompt bytes contribute zero gradient while response bytes do.

The prompt format is a single canonical layout shared byte-for-byte by training and inference, defined in one module so no other file may invent marker strings. A training example lays out a system block, a user block, an evidence block (one row per retrieved item carrying its source id, content, authority, and authenticity), an optional plan block, an assistant block, and an end marker. At inference the prompt ends at the assistant marker and the model continues from there; generation halts on the first of three stop sequences. When a byte budget forces truncation, the oldest turns are dropped first and the system, current user, evidence, and plan blocks are never truncated.

### 5.3 What the corpus looked like in the one real GPU session

The empirical anchor of this report is a single A100-40GB session on 2026-07-02, run by the owner on a cloud notebook. The validated corpus for that run was 0.367 GB across 142 shards with whole-document and per-repository holdout and all license checks passing: 151.9 MB of code, 120.0 MB of educational expository text, 54.8 MB of books (including 21 owner-committed public-domain texts totaling 23.5 MB alongside downloaded classics), and 40 MB of internal training-loop traces. The size is modest by design; the point of the session was to prove the pipeline end-to-end, not to saturate the model.

The one real GPU session used a validated 0.367 GB mixed corpus.

## 6. Engineering and System Implementation

### 6.1 The reasoning behind a hand-built, dual-backend system

The engineering choices follow from a single principle: an architectural claim is only as trustworthy as the code that realizes it and the ability to reproduce its results. That principle produced two backends. The primary backend builds every layer from parameter tensors and elementwise math on top of a mainstream framework used purely as an autograd and optimizer substrate — no pre-built transformer, attention, normalization, embedding, or state-space modules. Numerics are hardened for mixed-precision training: normalization statistics are computed in full precision even under half-precision autocast (squaring half-precision activations overflows), and masked softmax uses dtype-aware fill and denominator constants so a fully-masked row degrades to exact zeros instead of producing not-a-number values. A pair of GPU/mixed-precision tests gate any half-precision run.

The second backend is a from-scratch reverse-mode automatic differentiation engine implemented in NumPy at double precision. It records, for every operation, how to push gradients to its inputs, and runs a topological sort from the output in reverse; broadcasting is handled correctly by summing gradients back to their original shape. Double precision is a deliberate choice so that finite-difference gradient checks are clean. On this engine the project reimplements the same architecture, an optimizer, a chat loader, a held-out evaluation harness, and even the authority classifier — so the entire symbolic stack can run on NumPy alone, with no framework installed.

The payoff is a reproducibility property that is rare for a claimed defense result: a provenance benchmark's output reproduces **bit-identically** across operating systems (Windows to Linux), Python versions (3.13 to 3.11), and backends (the framework to the NumPy engine). Bit-identical cross-backend reproduction is strong evidence that the result is a property of the specification, not of a particular numerical environment.

### 6.2 Resumable, atomic, best-by-validation checkpointing

Training is designed to survive interruption. Checkpoints are written atomically, the best checkpoint is selected by validation rather than by last step, and training resumes from the exact shard and step with continuous optimizer momentum and variance state rather than a fresh optimizer. This is not a drill: during the 100K-conversation fine-tuning run the sandbox restarted partway through the fourth shard and killed the process; on relaunch the run reloaded the latest checkpoint, reported the correct resume point, and continued with the same optimizer state, and a test verifies in miniature that a resumed run reaches the same total steps and examples as an uninterrupted one.

The best-by-validation rule has teeth. In the one real fine-tuning session the trainer of that era saved only final weights, so the deployed checkpoint was the *overfit endpoint* rather than the validation optimum — a concrete demonstration of why best-by-validation, atomic saves, and early stopping were then made the default. A separate and painful operational lesson came from base pretraining: continued training reached a best held-out 1.313 bits per byte, and that checkpoint was **lost to an unpersisted runtime** because it lived only in volatile storage. The response was a 30-minute auto-backup mechanism. We report both losses because the reproducibility discipline is only credible if it includes the times it failed.

### 6.3 Training journals and the test suite

Every compute stage is scripted and journaled: notebook pipelines for a cloud A100 run and a second cloud platform mirror a desktop runbook cell for cell, so the same pipeline runs in three places and pushes its metrics back. A cloud analysis session and a local GPU session divide labor explicitly through committed reports.

The repository's test suite is substantial and is treated as a frozen contract on the neural core. At the time of writing, the test runner collects 249 tests (verified by direct collection; the count exceeds the number of test functions because parametrized cases expand at collection time, and it grows as regression tests accumulate). The suite specifically pins causality (that no position can attend to the future along any path), the response-only loss mask, checkpoint save/resume with continuous optimizer state, best-by-validation selection, generation termination at stop bytes, the prompt-format round-trip and drift guard, and the mixed-precision numerics. On a framework-free checkout, the torch-gated tests are skipped or fail only for the missing framework, while the NumPy-engine tests pass, which is what makes the second backend a genuine independent check rather than a mirror.

**In plain terms:** The whole system is built twice — once on a standard deep-learning toolkit and once from scratch in plain NumPy — and the two versions agree down to the last bit, which is a strong sign the results are real and not an accident of one computer. Training is built to survive crashes: it saves safely, keeps the best version rather than the last, and picks up exactly where it left off. That safety net was tested for real when a machine restarted mid-run, and it was strengthened after a good checkpoint was lost to a machine that forgot its files. A large automated test suite guards the core so that experiments cannot silently break it.

**Summary.** NUERONCE is implemented from primitives on two independent backends — a mainstream framework and a from-scratch NumPy double-precision autograd engine — that reproduce a provenance benchmark bit-for-bit across operating systems, Python versions, and backends; training is atomic, resumable, and best-by-validation, hardened by real interruption-and-loss incidents that added auto-backup; and a large test suite freezes causality, masking, checkpointing, and format invariants.

## 7. Results and Evaluation

### 7.1 Base pretraining: the one real scaling curve

The central positive result is the 2026-07-02 A100 session. Training an 11.1M-parameter model from scratch on the 0.367 GB mixed corpus at sequence length 256 and batch 64 with a learning rate of 5e-4 under half-precision autocast, held-out bits per byte fell from 4.515 to 1.785 in 7,524 steps and 55 minutes at a steady throughput of about 37,000 byte-targets per second. A pre-registered gate — held-out bits per byte below 1.8, written down as the precondition for starting fine-tuning — was cleared in-session at roughly step 7,000 (1.797) at minute 51, ahead of the prior one-to-two-session estimate. There were no crashes, and atomic best-checkpoint saves fired throughout, with the final best at 1.785 against a final latest of 1.787.

The qualitative milestones along the curve are worth recording because they show *what* the loss buys. Around 2.67 bits per byte, greedy decoding loops ("the the of the") and a code prompt yields only a newline. Around 2.11 bits per byte, sampled decoding produces morphologically legal invented words ("probacle," "packagragition") and English sentence rhythm. Around 2.03 bits per byte, a code prompt yields indented Python including a recognizable assertion call, and expository and narrative prose registers begin to separate. Continued training in a later session reached a best held-out 1.313 bits per byte after roughly 168 additional minutes, and it is around 1.31 that the first coherent greedy prose appeared — a sentence beginning "The nature of human understanding is the concept of many people and their communities. This course unit will explore the impor…". That checkpoint was then lost to an unpersisted runtime, as described in Section 6. We report the 1.313 figure and the emergence of coherent prose as observed, and we flag that the checkpoint itself no longer exists to re-evaluate.

A capacity observation from the same session: at the 0.367 GB corpus, 11M parameters is not yet the bottleneck — training bits per byte was around 1.2 against a held-out 1.785 with the curve still descending at cutoff, and the plateau was estimated at 1.6 to 1.75 after several more A100-hours. This is consistent with the earlier finding that base quality, not fine-tuning step count, is the binding constraint.

### 7.2 Supervised fine-tuning: a textbook overfitting case study

The fine-tuning stage in the same session is a clean negative-then-corrected result. A maximum length of 384 combined with an approximately 300-byte system prompt filtered roughly 20,000 human-assistant pairs down to just 429 examples (8 held out for validation). Validation loss bottomed at 1.246 at step 225 (about 1.6 minutes), then climbed to roughly 1.94 by step 2,425 while training loss fell to about 0.05 — textbook memorization. Because the trainer of that era saved only final weights, the deployed checkpoint was the overfit endpoint. A chat probe of that checkpoint produced the project's first turn-taking behavior from a real-corpus base — a greeting answered with a (garbled) greeting, "Hello! Whow w are you?" — while long out-of-distribution technical prompts returned empty responses, exactly as expected at 429 examples. The fixes were pushed the same session: raise the maximum length to 1,024 so pairs survive the filter, save a best-by-validation checkpoint with atomic saves, and early-stop after a fixed number of non-improving evaluations.

This episode is a small replication of the project's most-repeated lesson: fine-tuning *shapes* competence but cannot *create* it, and a weak base under fine-tuning improves its loss while producing unusable text.

### 7.3 The 100K-conversation fine-tuning study, at 112K parameters

An earlier, entirely NumPy-engine study fine-tuned a 112,709-parameter model on 100,000 synthetic conversations (held-out validation and test of 5,000 each, leakage-checked with zero overlaps). Held-out byte accuracy reached 89.06% on the test set, and the validation curve improved monotonically apart from one shard — a decisive fix of the overfitting seen in an earlier 61-example run. But the study's real value is its honesty about *what did not generalize*: arithmetic computation was memorized as structure, not learned as procedure. On novel operand pairs the strict check-pass rate was 2.1%, and even on exact training pairs used as memorization probes it was 4.7% — the model reliably produces the *shape* of an arithmetic answer ("36 times 52 equals …") while the *computed number* is almost never right, even for questions it was trained on. A separate metric-design lesson is reported against the project itself: a naive classification check showed an apparent 83% parity accuracy that collapsed to 1 of 24 once the check required the response to echo the same number it was asked about. The conclusion — structure generalizes before content at these scales — is why the runbooks insist on choice-ranking metrics alongside generative ones and forbid discarding checkpoints on generative exact-match alone.

### 7.4 The prompt-format drift incident

A single checkpoint from before the canonical format scored a teacher-forced byte accuracy of 0.906 when evaluated in its own (legacy) format and only 0.524 when evaluated under the mismatched canonical format — a silent 38-point regression that looked like a weak model but was purely an evaluation-format mismatch. This motivated stamping the prompt format into every checkpoint, resolving unstamped checkpoints to legacy, and a rule enforced by tests: never compare accuracies across formats and call the difference model quality.

### 7.5 The provenance/authority result

The strongest defensible result in the project is not from the byte model but from the provenance layer. A from-scratch, calibrated authority classifier (expected calibration error 0.004) holds **zero acceptance of poisoned documents across every text-borne attack family** — impersonation, laundering, citation-spoofing, paraphrase, and Unicode confusables — on a blind set generated by an independent code path, while naive baselines poison at rates of 0.75 to 1.00. The one residual failure under a feature-identical "perfect spoof" (0.278 acceptance) was proven to be an information-theoretic ceiling — such forgeries are unlearnable by construction — and then eliminated by adding an Ed25519 cryptographic provenance gate that separates apparent authority from authenticity, driving poison acceptance to zero with calibrated abstention rather than silent failure. The honest cost of removing the labels-given assumption was measured, not assumed: an oracle accuracy of 1.000 drops to a predicted 0.838 end-to-end, because a 0.994-accurate classifier's rare errors land on decisive documents. In the integrated evaluation, the full pipeline reaches 0.920 answer accuracy at 0.000 poison acceptance; ablating retrieval drops answer accuracy to 0.480 (retrieval is load-bearing), ablating provenance raises poison acceptance to 0.260, and ablating the verifier raises the unsupported-claim rate from 0.000 to 0.090. This whole benchmark is what reproduces bit-identically across environments.

Two orthogonal axes: apparent authority (low/medium/high, estimated from text by the learned classifier) and authenticity (verified/unverified/failed/revoked, decided by Ed25519 signature verification with content-hash integrity, issuer-key binding, expiry, and revocation). ### 7.6 A falsified hypothesis: typed-channel specialization

The architecture's differentiation story leaned heavily on the idea that the typed memory channels specialize into distinct roles. A probe trained a small model on tasks with distinct memory demands (local structure, long-range key recall, running count), then zeroed each channel and measured per-task loss. Removing *all* typed channels raised loss only on long-range recall (about 3.5%) and near-zero elsewhere; removing any *single* channel changed loss by less than 0.3% for every task. The channels behave as a diffuse, redundant pool that the dense read-out routes around, not as specialized roles. Per the project's own rule, the specialization hypothesis is recorded as **falsified at this scale and training**, and the differentiation story is instructed not to lean on typed channels until specialization is explicitly engineered and re-probed.

**In plain terms:** The one real training run learned to write fluent-looking English in under an hour and even started producing a genuinely coherent sentence a bit later — but that better version was accidentally deleted, which we admit. When we tried to teach it to chat on too few examples, it memorized them and got worse, exactly as the theory predicted, and we fixed the recipe. A separate study showed the model can copy the *look* of a math answer perfectly while getting the actual number wrong almost every time. The part that works best is the source-checking system, which refuses every faked document we throw at it. And the idea that the model's seven memory lanes each do a special job turned out to be wrong at this size, which we report as a failed prediction.

**Summary.** The trained 11.1M model reached 1.785 held-out bits per byte in 55 minutes and 1.313 later (that checkpoint lost), with fluency emerging along the curve; fine-tuning overfit as predicted and was corrected; a 112K-parameter study reached 89% held-out byte accuracy but exposed that arithmetic content does not generalize; a format-drift incident motivated checkpoint stamping; the provenance layer holds zero poison acceptance and reproduces bit-identically; and the typed-channel specialization hypothesis was falsified at this scale.

## 8. Discussion

### 8.1 Strengths, argued from the evidence

The most defensible strength is the reproducibility discipline itself. Bit-identical cross-backend, cross-operating-system, cross-Python reproduction of a full benchmark is uncommon, and it converts "our result" into "a property of the specification." The second strength is the provenance/authority layer: separating apparent authority from cryptographically-provable authenticity yields a system that cannot be poisoned by any text-only attack, with the residual attack surface reduced to key compromise and revocation measured rather than assumed. The third strength is process honesty — a hypothesis ledger that records one hypothesis strongly supported, two partially, one falsified, and several unstarted, and reports lost checkpoints and overfitting failures in full. In a field where negative results are often quietly dropped, this trail is itself a contribution, because it makes the positive claims trustworthy. The fourth strength is that the architecture demonstrably trains: from-scratch, a small byte model reached fluency-shaped output on a modest, license-clean corpus in under an hour, and the loss curve was still descending, suggesting the 11M rung was data-limited rather than capacity-limited at 0.367 GB.

### 8.2 Limits, stated plainly

The limits are as important as the strengths. First, **scale**: the largest trained model is 11.1M parameters; the 35M, 90M, and 337M configurations build and forward but have not cleared (or, for the largest, ever started) training. Any claim about NUERONCE's ceiling is therefore an extrapolation. Second, **the validation set is private and not externally comparable**: the 1.785 and 1.313 bits-per-byte figures are on the project's own held-out split of its own corpus, so they cannot be compared to published numbers on standardized corpora. Third, **there is no baseline face-off**: no equal-compute, equal-data plain transformer has been trained alongside NUERONCE, so we cannot claim the architecture beats a conventional design at matched budget — we can only claim it works. Fourth, **the two halves are islands**: the learned byte core and the symbolic provenance/cognitive layer are connected only by a data contract; the neural model never conditions on authority, and the authority classifier never feeds the byte model's generation. The strongest provenance result is thus a property of the symbolic layer, not of an integrated model. Fifth, **one architectural hypothesis is falsified** and others are untested, so the differentiation argument currently rests on provenance and process, not on the typed memory. Sixth, a specific benchmark caveat: a "full system equals contract-only system" tie in one provenance evaluation was guaranteed by construction, because that evaluation simulated the pipeline rather than running the real byte model — the right observation with the wrong implied cause, and a reason to be cautious about over-reading any single gate.

**In plain terms:** What is genuinely strong here is the honesty and the source-checking. The model itself is small, tested only against its own private yardstick, and never raced head-to-head against a conventional model of the same size — so we can say it works, not that it wins. And the clever source-checking part and the language part are still two separate systems bolted together, not one unified model. We are careful not to oversell any of it.

**Summary.** NUERONCE's credible strengths are its bit-identical reproducibility, its un-poisonable provenance layer, its process honesty, and a demonstrably trainable architecture; its honest limits are small trained scale, a private and non-comparable validation set, no equal-compute baseline, an unintegrated neural/symbolic split, a falsified core hypothesis, and at least one benchmark tie that was true by construction.

## 9. Assumptions and Future Work

The following are itemized so a reviewer can scrutinize each.

**Assumptions currently baked into the results.**
1. Held-out bits per byte on the project's own corpus is a meaningful quality proxy; this is standard for byte models but is not externally comparable.
2. The pre-registered gate of 1.8 bits per byte is a reasonable precondition for fine-tuning at the 11M rung; it is a project convention, not a derived threshold.
3. The synthetic conversational data is a fair proxy for instruction behavior at small scale; it trades organic diversity for zero licensing risk and exact reproducibility.
4. The provenance benchmark's attack families are representative of real text-borne attacks; the strongest claim is restricted to text-only attacks, with key compromise explicitly out of scope and handled by revocation.
5. The parameter-count labels (11.1M / 34.4M / 92.1M / 337M) are counted by construction and assumed to track effective capacity in the usual way.

**Future work, itemized.**
1. **Train the 35M rung to its acceptance criteria** — held-out bits per byte at or below 1.5, choice-ranking accuracy beating chance by at least 15 points on at least three multiple-choice subjects, at least 60% valid non-echo stop-terminated answers, and five of five grammatical sample transcripts — then unlock the 90M rung, and only then consider the 337M configuration.
2. **Run a standardized external benchmark** (for example a byte-level compression benchmark such as enwik8 or text8) so the bits-per-byte numbers become externally comparable. None has been run.
3. **Run an equal-compute, equal-data plain-transformer baseline** so the hybrid routed core can be judged against a conventional design at matched budget. None has been run.
4. **Integrate the neural core and the provenance layer** so the byte model conditions on authority-gated evidence and the classifier feeds generation, replacing the current data-contract bridge and closing the "two islands" gap.
5. **Re-engineer and re-probe typed-channel specialization** with per-channel read-out heads or channel-specific auxiliary losses, since the diffuse-pool result falsified the hoped-for specialization.
6. **Replace the simulated provenance evaluation with the real learned pipeline in the loop**, and add an externally-authored naturalistic blind benchmark so the provenance claim can survive outside review.
7. **Recover or retrain the lost 1.313-bits-per-byte checkpoint** now that auto-backup exists, and confirm the coherent-prose emergence reproducibly.
8. **Complete the later fine-tuning stages** (verifier-guided revision, residual correction experts, controlled continual learning) that are currently deliberate scaffolding, once a base model worth revising exists.

**In plain terms:** The next steps are clear and honest: train the bigger versions to hit targets set in advance, finally measure the model on a public yardstick and against a normal model of the same size, and stitch the language part and the source-checking part into one system. We also want to rebuild the good checkpoint we lost and fix the memory-lane idea that did not pan out.

**Summary.** The results rest on a handful of explicit, reasonable assumptions, and the roadmap is a prioritized ladder: prove the 35M rung against pre-registered gates, add external and baseline comparisons, integrate the two subsystems, redesign the falsified component, and finish the deferred training stages.
